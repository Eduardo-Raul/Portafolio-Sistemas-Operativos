# Comunicación mediante tuberias


La comunicacion entre procesos es fundamental para poder intercambiar datos, para poder comunicar procesos, debemos considerar si estos se van a comunicar en la misma maquina o si estan en diferentes maquinas


Las tuberías son mecanismos de comunicación entre dos o mas procesos en la misma maquina en sistemas UNIX System V

Su funcionamiento es muy parecido al de una tubería real, pues un proceso escribe información por un extremo y otro proceso la recibe del otro lado, es decir, un proceso genera información y otro proceso la muestra


`Proceso A ----> [Tubería] ----> Proceso B`

- El proceso A escribe datos
- La tubería transporta los datos
- El proceso B los lee o los muestra


---



# Tuberias sin nombre 


Las tuberías sin nombre, tambien llamadas `pipes` son mecanismos de comunicación proporcionados por todos los sistemas UNIX, se llaman anonimas porque:

- no tienen nombre dentro del sistema de archivos
- solo existen mientras los procesos están ejecutandose

Su facilidad de uso les da una ventaja para implementarlas de manera sencilla, pero presentan dos limitaciones importantes:

1. Los datos solo fluyen en una dirección.
2. Pueden usarse solo entre procesos que tengan un ancestro en común

Es decir, normalmente la tuberia es creada por un proceso que invoca a la funcion `fork()` y es usada entre Padre --> hijo, con una comunicación unidireccional, es decir en un solo sentido van a viajar los datos, si se quisiera una comunicacion en ambos sentidos se tendrian que ocupar dos tuberias 

Las tuberias sin nombre se crean llamando a la función `pipe()`


---

PROTOTIPO DE LA FUNCIÓN
```c
#include <unistd.h>

int pipe (int filedes[2]);

int pipe2 (int filedes[2], int flags);
```


---


El argumento filedes son los descriptores del archivo y se utilizan para asignar la lectura y escritura, donde:

- filedes[0] es para lectura
- filedes[1] es para escritura


La función `pipe2()` es una version más moderna que nos permite configurar ciertas opciones desde el momento que se crea la tuveria, algunas opciones de `flags` son:

- 0: Actua de forma natural como `pipe()`
- O_CLOEXEC: Cierra automaticamente los descriptores de archivo al ejecutar `exec()`
- O_DIRECT: Realiza la entrada/salida en modo "paquete", es decir, cada escritura y lectura en la tuberia se procesa como un paquete independiente
- O_NONBLOCK: Modo no bloqueante, si no hay datos que leer, la funcion regresa inmediatamente con un error `EAGAIN`

---


## Como escribir en una tuberia 


Para poder escribir datos y mandarlos por una tuberia, al igual que para leerlos, se deben considerar algunas cosas, primero que nada, las funciones que nos ayudan a la escritura y lectura son `write()` y `read()` donde:

- `read()` recibe la infrmación que se envía 
- `write()` escribe y manda la información por la tuberia


---


PROTOTIPO DE LA FUNCIÓN
```c
#include <unistd.h>

ssize_t write(int fd, const void *buf, size_t count);

ssize_t read(int fd, void *buf, size_t count);
```


---


Los parametros de `write()` son:
- fd: dónde escribir
- buf: datos a enviar
- count: cantidad de bytes

entonces si tenemos write(fd[1], mensaje, strlen(mensaje)+1);

- fd[1] indica escribir en la tuberia
- mensaje son los datos que se envian
- strlen(mensaje)+1 indica cuantos bytes se van a enviar (el +1 es para incluir el caracter final `\0`)


Los parametros de `read()` son:
- fd: de dónde leer
- buf: dónde guardar los datos
- count: cantidad máxima de bytes a leer

entonces teniendo lo siguiente
```c
char buffer[100];
read(fd[0], buffer, sizeof(buffer));
```

- fd[0] indica leer desde la tuberia
- buffer es el espacio donde se guardarán los datos leídos
- sizeof(buffer) indica el numero máximo de bytes que se pueden leer


Aunque algo a resaltar es que `read()` devuelve un numero entero:

- `>0` para indicar la cantidad de bytes leidos
- 0 para indicar fin del archivo o pipe cerrado
- -1 para indicar que ocurrió un error

## Codigo de ejemplo

En el siguiente ejemplo veremos la creacion de una tuberia sin nombre y la comunicación entre un proceso padre e hijo, en donde el hijo envia un mensaje y su padre lo recibe e imprime en pantalla, al igual que vemos como tambuen es posible ocupar el valor de retorno de la función `read()`

Algo importante de mencionar es que se debe cerrar el descriptor que no se este utilizando para evitar errores, es decir:

- si un proceso va a leer debe cerrar la escritura
- si un proceso va a escribir debe cerrar la lectura


---


```c
#include <stdio.h>
#include <stdlib.h>
#include <sys/types.h>
#include <unistd.h>
#include <sys/wait.h>
#define MAXLINE 80

int main()
{
    int n, fd[2];
    pid_t hijo;
    char linea[MAXLINE];
    int estado;
    if (pipe(fd) < 0)
    {
        fprintf(stderr, "error de tubería");
        exit(EXIT_FAILURE);
    }
    if ((hijo = fork()) == 0)
    {
        close(fd[0]);
        write(fd[1], "hola mundo \n", 12);
    }
    else
    {
        close(fd[1]);
        n = read(fd[0], linea, MAXLINE);
        write(STDOUT_FILENO, linea, n);
        printf("numero de lineas %d \n", n);
    }
    return EXIT_SUCCESS;
}
```


---


## Salida en pantalla
hola mundo 
numero de lineas 12

## Aprendizaje y mejoras


En este tema vimos la creación y el funcionamiento de las tuberías, como es posible comunicar procesos para mandar datos entre ellos, algo muy importante para la programación concurrente y las medidas que se tienen que considerar para evitar bloqueos o problemas la momento de comunicar procesos

Una propuesta para mejorar lo aprendido en la practica seria la implementacion de comunicaciones bidireccionales con el uso de tuberias extra para demostrar como es posible que tanto el proceso padre como el hijo pueden recibir y mandar datos entre ellos, al igual que combinar esto con el tema de los hilos para poder aprovechar al maximo la programación recurrente y procesar multiples tareas al mismo tiempo
